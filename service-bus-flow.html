Service Bus Flow

AuctionService owns the write model in PostgreSQL. SearchService keeps a denormalized MongoDB read model in sync by consuming events from RabbitMQ.

AuctionService

Controller action

Create, update, or delete auction data through the API.

Save to PostgreSQL

The auction database remains the source of truth.

Publish event

MassTransit sends a contract message after the database change succeeds.

RabbitMQ

Message broker

The service bus decouples AuctionService from SearchService.

SearchService

Consume event

MassTransit routes each message to the matching consumer class.

Update MongoDB

The search index is created, patched, or deleted based on the event.

Search stays fresh

Users query the MongoDB read model without calling AuctionService directly.

API request arrives A user creates, updates, or deletes an auction.
Database change succeeds AuctionService saves the change in PostgreSQL first.
Event is published MassTransit publishes the matching contract to RabbitMQ.
Consumer receives it SearchService handles the message with the matching consumer.
Search model updates MongoDB is synchronized for fast search queries.
